### 阶段2: 观察（建议执行）🎯

**目标**: 在新开发中自然修复

**策略**: **遇到就改，不主动找**

#### 执行规则

**✅ 什么时候修复**:
1. 修改某个文件时，顺便统一该文件的参数
2. 新增功能时，使用标准参数名
3. 修复bug时，涉及的代码统一参数
4. 重构某个模块时，统一该模块的参数

**❌ 什么时候不修复**:
1. 文件运行正常，没有修改需求
2. 代码很复杂，不确定影响范围
3. 时间紧急，需要快速修复
4. 备份文件、测试文件、废弃代码

#### 修复步骤（安全）

```bash
# 1. 修改前备份
git add -A
git commit -m "🔄 修改前备份: 准备统一[文件名]的参数"

# 2. 小范围修改（一个文件或一个功能）
# 例如：只修改 index_smart_workflow.html

# 3. 本地测试
# - 打开页面，测试所有功能
# - 检查控制台无错误
# - 验证API调用正常

# 4. 运行检查
python3 scripts/parameter_consistency_enforcer.py

# 5. 如果出问题，立即回滚
git reset --hard HEAD^

# 6. 如果没问题，提交
git add -A
git commit -m "✨ 统一[文件名]的参数命名"
```

**风险**: 🟡 低风险（小范围，可回滚）

## 🛡️ 安全保障措施

### 1. Git备份策略

```bash
# 修改任何代码前，先备份
git add -A
git commit -m "🔄 修改前备份: [描述即将做的修改]"

# 这样如果出问题可以立即回滚
git reset --hard HEAD^
```

### 2. 测试清单（修改后必须测试）

```
修改后测试清单:
□ 页面能正常打开
□ 功能能正常使用
□ 控制台无错误
□ API调用成功
□ 数据显示正确
□ 移动端/PC端都正常
```

### 3. 回滚流程

```bash
# 如果发现问题，立即回滚
git reset --hard HEAD^

# 或者回滚到更早的版本
git log --oneline  # 查看历史
git reset --hard <commit-id>
```
