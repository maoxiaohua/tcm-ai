# 双向连接支持修复

## 🐛 问题发现

### 用户反馈
用户反馈增量追加功能"还是不行"，通过控制台日志发现了真正的问题：

**控制台日志**：
```
🔗 开始同步连接关系
  从: 处方2 (prescription)
  到: 脑袋瓜前面疼 (syndrome_branch)
  操作: 创建连接
💡 未找到匹配的证候-处方关系，可能需要手动更新
```

### 问题根源分析

**发现问题**：连接方向是 `处方 → 证候`，但代码只检查 `证候 → 处方`！

**原有代码逻辑**（第2336行）：
```javascript
if ((fromNode.type === 'syndrome' || fromNode.type === 'syndrome_branch') && toNode.type === 'prescription') {
    // 同步逻辑
}
```

**实际连接情况**：
```
用户操作：从"处方2"拖拽连接线到"脑袋瓜前面疼"
实际方向：处方2 → 脑袋瓜前面疼（处方 → 证候）
代码检查：fromNode.type === 'prescription' ❌ 不匹配！
```

**为什么会有反向连接？**
- 画布上用户可以从任意节点向任意节点拖拽连接线
- 用户习惯：有时从证候拖到处方，有时从处方拖到证候
- 医学逻辑上都是"证候对应处方"，但画布操作上方向可以任意

## ✅ 解决方案

### 核心思路：支持双向连接

无论用户从证候拖到处方，还是从处方拖到证候，都应该正确识别并同步为"证候的处方列表"。

### 实现方式：自动交换节点

**位置**：`/opt/tcm-ai/static/decision_tree_visual_builder.html` 第2404-2475行

**新增代码**：
```javascript
// 场景：处方 → 证候 的连接（反向连接，需要交换节点）
else if (fromNode.type === 'prescription' && (toNode.type === 'syndrome' || toNode.type === 'syndrome_branch')) {
    console.log('🔄 检测到反向连接（处方→证候），自动交换节点处理...');

    // 交换节点，按 证候→处方 的逻辑处理
    const actualSyndrome = toNode;      // 实际的证候节点（原toNode）
    const actualPrescription = fromNode; // 实际的处方节点（原fromNode）

    // 后续逻辑完全相同，只是使用交换后的节点变量
    // ... （完整的增量追加和智能删除逻辑）
}
```

### 技术细节

**检测条件**：
```javascript
fromNode.type === 'prescription' && (toNode.type === 'syndrome' || toNode.type === 'syndrome_branch')
```

**节点交换**：
```javascript
const actualSyndrome = toNode;        // 证候节点
const actualPrescription = fromNode;   // 处方节点
```

**逻辑复用**：
- 完全复用正向连接的增量追加逻辑
- 完全复用正向连接的智能删除逻辑
- 只是变量名从 `fromNode/toNode` 改为 `actualSyndrome/actualPrescription`

## 💡 设计思考

### 为什么不限制连接方向？

**方案A**：强制只能从证候拖到处方
- ❌ 限制用户操作自由度
- ❌ 增加学习成本
- ❌ 容易误操作后困惑

**方案B**：支持双向连接，自动识别（✅ 当前方案）
- ✅ 用户操作自由
- ✅ 智能处理，无需学习规则
- ✅ 符合直觉

### 代码复用 vs 代码抽取

**当前实现**：两个if-else分支，代码重复
```javascript
if (正向连接) {
    // 增量追加逻辑
    // 智能删除逻辑
} else if (反向连接) {
    // 交换节点
    // 增量追加逻辑（重复）
    // 智能删除逻辑（重复）
}
```

**优化方向**（未来可考虑）：
```javascript
// 统一节点变量
let syndromeNode, prescriptionNode;

if (正向连接) {
    syndromeNode = fromNode;
    prescriptionNode = toNode;
} else if (反向连接) {
    syndromeNode = toNode;
    prescriptionNode = fromNode;
} else {
    return; // 不支持的连接类型
}

// 统一的处理逻辑（只写一次）
// ...
```

**为什么当前不优化？**
- ✅ 代码清晰，易于理解
- ✅ 修改风险低（独立分支）
- ✅ 性能影响可忽略（只执行一次）
- ⏳ 待功能稳定后再重构
