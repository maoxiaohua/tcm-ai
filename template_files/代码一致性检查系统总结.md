## 🔍 功能特性

### 7大检查类别

| # | 检查类别 | 优先级 | 检查内容 | 覆盖范围 |
|---|---------|-------|---------|---------|
| 1 | **API接口一致性** | ⭐⭐⭐ | 前后端API是否匹配 | 154个后端API, 94个前端调用 |
| 2 | **参数命名一致性** | ⭐⭐⭐ | 参数名称是否统一 | user_id, doctor_id等4组参数 |
| 3 | **数据库字段引用** | ⭐⭐ | SQL查询字段是否存在 | 68张表, 数百个字段 |
| 4 | **数据结构一致性** | ⭐⭐ | 前后端数据格式是否匹配 | 响应格式、数据访问路径 |
| 5 | **常见错误模式** | ⭐⭐ | 检测已知错误模式 | 4种错误模式 |
| 6 | **未使用API检测** | ⭐ | 发现冗余API | 后端API使用率分析 |
| 7 | **数据库Schema** | ⭐ | 表结构完整性 | 2个数据库, 68张表 |

### 核心能力

✅ **自动化检查**: 无需手动操作，Git提交时自动运行
✅ **全面覆盖**: 7大检查类别，覆盖API、数据库、前后端
✅ **智能分析**: 路径参数匹配、别名识别、模糊匹配
✅ **友好报告**: 终端彩色输出 + JSON详细报告 + 交互式查看器
✅ **问题分级**: 错误/警告/信息三级分类
✅ **易于扩展**: 清晰的代码结构，易于添加新规则

## 🚀 使用方式

### 方式1: 手动检查（推荐）

```bash
# 运行完整检查
cd /opt/tcm-ai
python3 scripts/consistency_checker.py

# 查看友好报告
bash scripts/view_consistency_report.sh
```

### 方式2: Git自动检查

```bash
# 正常提交（自动检查）
git add .
git commit -m "修改说明"

# 系统会运行：
# [1/2] 基础功能检查 (26项)
# [2/2] 代码一致性检查 (7类)
```

### 方式3: 查看已有报告

```bash
# 查看JSON报告
cat template_files/consistency_check_report.json | python3 -m json.tool

# 使用交互式查看器
bash scripts/view_consistency_report.sh
```

## 🔧 高级功能

### 自定义检查规则

可在 `scripts/consistency_checker.py` 中添加：

```python
error_patterns = [
    {
        'name': '自定义规则',
        'pattern': r'正则表达式',
        'message': '错误提示'
    }
]
```

### 过滤已知问题

```python
KNOWN_ISSUES = [
    '/api/some-old-endpoint',  # 计划废弃
]

if api_path in KNOWN_ISSUES:
    continue  # 跳过检查
```

### 调整严格度

```python
# 宽松模式：只报告明确错误
# 严格模式：所有不匹配都报错
```

## 📈 改进建议

基于首次检查结果，建议：

### 短期 (1-2周)

1. **修复关键API错误** (优先级: P0)
   - 修复前端调用但后端不存在的API
   - 重点: `/api/v2/auth/*` 系列端点

2. **统一参数命名** (优先级: P1)
   - 统一使用 `user_id` 而非 `patient_id`
   - 统一使用 `doctor_id` 而非 `selected_doctor`

### 中期 (1个月)

3. **修复数据库字段引用** (优先级: P1)
   - 检查并修复所有字段名错误
   - 特别注意单复数形式

4. **统一响应格式** (优先级: P2)
   - 确保所有API使用统一的响应格式
   - 前端统一数据访问方式

### 长期 (持续)

5. **清理未使用API** (优先级: P3)
   - 评估112个未使用的后端API
   - 决定保留、废弃或补充前端调用

6. **持续监控** (优先级: P2)
   - 每周运行检查
   - 定期审查报告
   - 防止新问题引入
