# TCM-AI 代码一致性检查系统使用指南

## 📋 系统概述

本系统旨在**全面检查代码的一致性**，防止因新增或修改代码导致的API接口、数据库接口、前后端关联关系不一致而产生的bug。

### 核心价值
- ✅ **防止"修A坏B"**: 修改代码前自动检查影响范围
- ✅ **API一致性保证**: 确保前后端API调用匹配
- ✅ **数据库字段校验**: 避免引用不存在的字段
- ✅ **参数命名规范**: 统一参数命名，减少混淆
- ✅ **常见错误预防**: 自动检测已知的错误模式

## 🚀 快速开始

### 1. 手动运行检查

```bash
# 进入项目目录
cd /opt/tcm-ai

# 运行一致性检查
python3 scripts/consistency_checker.py

# 查看详细报告
cat template_files/consistency_check_report.json | python3 -m json.tool
```

### 2. Git提交时自动检查

```bash
# 正常提交（自动触发检查）
git add .
git commit -m "修复xxx功能"

# 如果检查失败，根据提示修复问题
# 修复后重新提交

# 紧急情况下跳过检查（不推荐）
git commit --no-verify -m "紧急修复"
```

### 3. 集成到开发流程

```bash
# 开发前：了解现有架构
python3 scripts/consistency_checker.py

# 开发中：随时检查
python3 scripts/consistency_checker.py

# 提交前：确保一致性
git add .
git commit -m "新增xxx功能"  # 自动运行检查
```

## 🔍 检查项目详解

### 1. API接口一致性检查 ⭐⭐⭐

**检查内容**:
- 前端调用的API是否在后端定义
- 后端定义的API是否被前端使用
- API路径参数是否匹配

**示例问题**:
```
❌ 错误: 前端调用的API不存在于后端: /api/chat
   调用文件: index_smart_workflow.html

✅ 修复: 使用正确的端点 /api/consultation/chat
```

**检查原理**:
1. 扫描 `api/routes/*.py` 提取所有后端路由定义
2. 扫描 `static/**/*.html` 和 `static/**/*.js` 提取fetch调用
3. 对比前后端API列表，标记不匹配项

### 2. 参数命名一致性检查 ⭐⭐⭐

**检查内容**:
- 参数名称是否统一（user_id vs userId vs patient_id）
- 是否使用了非标准的参数别名
- 参数格式是否一致（数字ID vs 字符串名称）

**示例问题**:
```
⚠️  警告: API参数命名不一致: /api/consultation/chat
   - 使用 'selected_doctor', 建议 'doctor_id'
   - 使用 'conversationId', 建议 'conversation_id'
```

**常见参数别名映射**:
| 标准名称 | 常见别名 |
|---------|---------|
| user_id | userId, patient_id, patientId |
| doctor_id | doctorId, selected_doctor, doctor_name |
| conversation_id | conversationId, session_id, sessionId |
| message | content, msg, text |

### 3. 数据库字段引用检查 ⭐⭐

**检查内容**:
- SQL查询中使用的字段是否存在于表中
- 字段名拼写是否正确
- 单复数形式是否匹配

**示例问题**:
```
❌ 错误: 字段不存在: doctors.uuid
   文件: unified_auth_routes.py

✅ 修复: 使用正确的字段名 doctors.id
```

**检查方法**:
1. 读取SQLite数据库schema
2. 扫描Python代码中的SQL查询
3. 验证查询字段是否存在于表结构中

### 4. 常见错误模式检查 ⭐⭐

**检查内容**:
- 错误的API端点（缺少 /api 前缀）
- 医生ID格式错误（数字 vs 名称）
- 响应数据结构访问错误
- try-catch块不匹配

**示例问题**:
```
⚠️  警告: 错误的API端点: index_smart_workflow.html
   使用了错误的API端点，应使用完整路径如 /api/consultation/chat

⚠️  警告: 医生ID格式错误: doctor_selection.js
   医生ID应使用名称格式(zhang_zhongjing)而非数字
```

### 报告结构

```json
    "warnings": 12,   // 警告：建议修复
    "info": 3         // 信息：仅供参考
  },
  "details": {
    "errors": [...],
    "warnings": [...],
    "info": [...]
  }
}
```

### 问题优先级

| 级别 | 说明 | 处理方式 |
|------|------|---------|
| **错误** | 严重问题，会导致功能失效 | 🔴 必须立即修复 |
| **警告** | 潜在问题，可能导致bug | 🟡 建议修复 |
| **信息** | 统计信息，无需处理 | 🟢 仅供参考 |

### 典型错误案例

#### 案例1: API端点不存在
```
[错误] [API一致性] 前端调用的API不存在于后端: /api/doctor/list
调用文件: index_smart_workflow.html
```

**根本原因**: 后端路由是 `/api/doctors/list`（复数），前端写错了

**修复方案**:
```javascript
// ❌ 错误
fetch('/api/doctor/list')

// ✅ 正确
fetch('/api/doctors/list')
```

#### 案例2: 参数格式不匹配
```
[警告] [参数命名] API参数命名不一致: /api/consultation/chat
使用 'selected_doctor', 建议 'doctor_id'
```

**根本原因**: 前端使用 `selected_doctor`，后端期望 `doctor_id`

**修复方案**:
```javascript
// 方案1: 修改前端参数名
fetch('/api/consultation/chat', {
  body: JSON.stringify({
    doctor_id: selectedDoctor  // 改为 doctor_id
  })
})

// 方案2: 添加参数映射（兼容性更好）
const params = {
  doctor_id: formData.selected_doctor || formData.doctor_id
}
```

#### 案例3: 数据库字段不存在
```
[错误] [数据库字段] 字段不存在: doctors.specialties
文件: doctor_routes.py
```

**根本原因**: 表中字段是 `speciality`（单数），代码中写成了 `specialties`（复数）

**修复方案**:
```python
# ❌ 错误
cursor.execute("SELECT specialties FROM doctors WHERE id = ?", (doctor_id,))

# ✅ 正确 - 先检查表结构
# sqlite3 database.sqlite ".schema doctors"
cursor.execute("SELECT speciality FROM doctors WHERE id = ?", (doctor_id,))
```

## 🛠️ 高级用法

### 1. 自定义检查规则

编辑 `scripts/consistency_checker.py`，在 `check_common_error_patterns()` 方法中添加：

```python
error_patterns = [
    # ... 现有规则 ...
    {
        'name': '自定义规则名称',
        'pattern': r'正则表达式',
        'message': '错误提示信息'
    }
]
```

### 2. 过滤特定警告

如果某些警告是已知且不需要修复的，可以在检查逻辑中添加过滤：

```python
# 在 check_api_consistency() 中
if not matched:
    # 过滤特定API
    if be_api in ['/api/health', '/api/internal/debug']:
        continue

    # 添加警告
    self.add_warning(...)
```

### 3. 生成HTML报告

可以扩展报告生成功能，输出更友好的HTML格式：

```python
# 在 generate_report() 中添加
html_report = self._generate_html_report()
html_path = PROJECT_ROOT / "template_files" / "consistency_report.html"
html_path.write_text(html_report, encoding='utf-8')
```

## 🔧 故障排除

### 问题1: 检查脚本运行失败

**症状**: `python3 scripts/consistency_checker.py` 报错

**可能原因**:
- Python版本过低（需要3.7+）
- 缺少依赖库
- 文件路径错误

**解决方案**:
```bash
# 检查Python版本
python3 --version  # 需要 >= 3.7

# 检查项目路径
cd /opt/tcm-ai
pwd

# 查看详细错误
python3 scripts/consistency_checker.py 2>&1 | tee check.log
```

### 问题2: 检查报告找不到

**症状**: 报告文件不存在

**解决方案**:
```bash
# 确保目录存在
mkdir -p template_files

# 重新运行检查
python3 scripts/consistency_checker.py

# 查看报告
ls -la template_files/consistency_check_report.json
```

### 问题3: Git hook不生效

**症状**: 提交时没有运行检查

**解决方案**:
```bash
# 检查hook文件是否存在
ls -la .git/hooks/pre-commit

# 添加执行权限
chmod +x .git/hooks/pre-commit

# 测试hook
bash .git/hooks/pre-commit
```

### 问题4: 大量误报

**症状**: 检查报告中有很多误报的问题

**解决方案**:
1. 分析误报模式
2. 在检查脚本中添加过滤规则
3. 调整正则表达式的匹配精度
4. 提交改进建议
